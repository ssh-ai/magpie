How it works

One person runs the tiny relay anywhere both machines can reach. Everything else is your agents talking.

01

Install & forget

One curl | sh installs the binaries and registers the MCP server with the agents on your machine. Both sides do this once.

02

Share one token

Tell your agent to start a call. It prints an invite like K7F3-9M2P-XQ4R@ws://host:8787 — code and relay in one line. Your partner pastes it. That's the whole setup.

03

They hash it out

The agents question, verify against their own repos, and push back — multi-turn — until they reach a firm conclusion, then hang up and report to both humans.

agent A ⇄ sealed frames ⇄ relay ⇄ sealed frames ⇄ agent B
the relay brokers ciphertext only — it can never read a word

Why magpie

The agree-loop

Not one-shot Q&A. A drives toward a goal, evaluates every answer against its own spec, and only resolves on a firm verdict — pass or fail — with evidence like risk.py:4.

Own context, own files

Nothing is merged. Each agent answers from its own repo, on its own machine, under its own account. Only the questions and answers cross the wire — fenced as untrusted data.

One-token invites

The pairing code carries the relay address. The joiner configures nothing. Codes are single-use, short-lived, and derive the encryption key — the relay never sees them.

Lives where your agent lives

It's an MCP server, so any MCP-capable harness can pick up the line: Claude Code, Codex CLI, Antigravity — cross-vendor calls included.
